MOTEL Platform

Methodology for Open Technology Data in Energy Models

MOTEL provides a reproducible workflow for preparing, harmonising, documenting, and reusing technology data in energy system models, while creating a clean handoff into ontology, graph database, API, and web application layers.

Full methodology GitHub repository

What is MOTEL for?

Energy system models rely on many technology assumptions such as CAPEX, efficiency, lifetime, emissions factors, and carrier relations. Those assumptions are often buried in spreadsheets or model-specific files, which makes them hard to trace, compare, and reuse. MOTEL addresses that by introducing a structured workflow for turning raw technology information into harmonised, provenance-aware, and ontology-ready data.

Traceable parameters

Technology values stay connected to source references, assumptions, and harmonisation history.

Reusable structures

Shared schemas, controlled vocabularies, and linked entities make the data easier to compare across studies and tools.

Ontology-aware handoff

Structured MOTEL outputs can feed a downstream ontology, GraphDB, backend API, and web interface.

Workflow

The MOTEL workflow connects data preparation with semantic representation and model-ready export.

  • Traceable technology parameters
  • Clear source documentation
  • Ontology-supported interoperability
  • Reusable data structures
  • Reproducible data preparation workflow
  • Model-ready export as a downstream goal

Methodology

  1. Data ingestion: raw or semi-structured technology data is collected and stored as unmapped records.
  2. Harmonisation: staged records are transformed into a consistent MOTEL data structure.
  3. Entity linking: technologies, parameters, carriers, scopes, and sources are represented through linked records and registries.
  4. Ontology mapping: harmonised MOTEL outputs can be connected to ontology classes and properties in a downstream repository.
  5. Application and export: the graph database, backend API, and webapp support exploration and future model-ready export workflows.

Repository Structure

The current repository focuses on the implemented data-preparation core. A second repository carries the ontology, GraphDB, backend, and frontend stack. 

motel-platform/
|-- 1_ingest/         source-specific ingestion notebooks and helpers
|-- 2_harmonise/      harmonisation notebooks and helper functions
|-- motel-db/         unmapped data, registries, mappings, and linked outputs
|-- schema/           machine-readable data schemas
|-- schema_simple/    human-readable schema blueprints
`-- docs/             GitHub Pages documentation and workflow figures

Example Technology Record

A harmonised MOTEL record can connect a technology, its parameters, units, source references, and ontology-ready identifiers in one inspectable structure. 

technology:
  id: heat_pump
  label: Heat pump
  type: EnergyConverter
  parameters:
    - id: heat_pump_capex
      label: CAPEX
      value: 1200
      unit: CHF/kW
      source_id: SRC_001
      ontology_class: dici_onto:CAPEX
    - id: heat_pump_lifetime
      label: Lifetime
      value: 25
      unit: years
      source_id: SRC_001
      ontology_class: dici_onto:Lifetime

Ontology and Graph Database Layer

The downstream MOTEL ontology repository turns harmonised MOTEL data into ontology-based TTL files, loads them into GraphDB, serves them through a backend API, and exposes them in a web interface for exploration. 

Heat pump
-> hasAttribute
-> CAPEX
-> hasValue
-> 1200 CHF/kW
-> wasDerivedFrom
-> Source record

Webapp and Model Builder

In the separate downstream stack, the MOTEL webapp is intended to let users browse technologies, inspect parameters and provenance, adapt selected values, and export model-ready configuration files.

  • Technology selection
  • Parameter table
  • Source and provenance display
  • Editable or customisable values
  • YAML export

How to Run

This repository can be explored locally with the notebook-based ingestion and harmonisation workflow. The downstream ontology, KG, backend, and frontend stack runs from a separate repository. 

git clone https://github.com/BartonChenTW/motel-platform
cd motel-platform
python -m venv .venv
# Windows PowerShell:
# .\.venv\Scripts\Activate.ps1
pip install -r requirements.txt

Then open the Step 1 and Step 2 notebooks locally in Jupyter or VS Code:

  • 1_ingest/1_data_ingestion.ipynb
  • 2_harmonise/2_data_harmonisation.ipynb

Outputs and Deliverables

  • A structured technology data model

  • Linked-entity target structures
  • A workflow handoff into ontology, graph, API, and web-app layers

Status and Limitations

Current Status

  • Data ingestion and harmonisation workflow implemented
  • Unmapped and linked data structures defined
  • Controlled vocabularies and mapping tables generated
  • Workflow handoff to ontology and graph layers documented

Current Limitations

  • The public repo currently demonstrates a selected set of technologies
  • Ontology mapping, GraphDB, backend, and frontend are in a separate repository
  • Model-ready YAML export is a downstream target, not a full feature of this repo alone
  • Additional validation and richer export formats remain future work
